# Функциональные требования

## 1. Контекст проекта
Цель проекта — анализ и фильтрация HTTP(S)-трафика веб‑приложений для выявления характерных паттернов атак (в т.ч. SQL‑инъекций и XSS) и снижения риска их успешной эксплуатации. Проект будет представлять из себя собственный reverse‑proxy + WAF‑движок:
- Реализация прокси на Go с middleware‑цепочкой: parse → normalize → detect → decision.
- Хранилище событий: ClickHouse (для аналитики).
- Веб‑панель: отдельный сервис (React) + API (Go/FastApi).


## 2. Границы системы
### Входит в систему
- Приём и проксирование HTTP(S) запросов к защищаемому приложению.
- Нормализация и анализ запросов (и при необходимости ответов).
- Принятие решения: пропуск/блокировка/логирование.
- Хранение событий.
- Веб‑панель для настройки правил и просмотра инцидентов.

## 3. Функциональные требования (FR)

### FR‑01. Проксирование и маршрутизация
- Система должна работать как reverse‑proxy (или явный forward‑proxy) и пропускать **весь** HTTP‑трафик к одному или нескольким backend‑сервисам.
- Система должна уметь проксировать HTTP/1.1; желательно — HTTP/2 (как минимум со стороны клиента или к backend).

### FR‑02. TLS и режимы развёртывания
- TLS termination на прокси (прокси принимает HTTPS и ходит к backend по HTTP/HTTPS).

### FR‑03. Захват и разбор HTTP данных
- Система должна извлекать для анализа: метод, URI, query‑string, заголовки, cookies, тело запроса (с ограничениями по размеру), IP/порт, User‑Agent, Referer, Content‑Type.
- Система должна корректно обрабатывать: URL‑encoding, двойное кодирование, base64‑поля (опционально), multipart/form‑data, JSON, XML, application/x-www-form-urlencoded.
- Система должна поддерживать ограничение максимального размера тела для анализа и отдельное ограничение для логирования (для приватности).

### FR‑04. Нормализация (каноникализация)
- Перед детектом система должна применять нормализацию: декодирование %, унификация регистра (для отдельных токенов), нормализация путей, удаление/сжатие повторяющихся разделителей, выявление обфускаций.
- Система должна сохранять оригинальные данные и нормализованное представление (для расследований).

### FR‑05. Детект SQL‑инъекций
- Система должна выявлять SQLi минимум по двум классам методов:
  - Сигнатурный детект (паттерны: UNION SELECT, OR 1=1, комментарии, stacked queries, time‑based, error‑based, boolean‑based).
  - Эвристический/аномальный скоринг (например: доля спецсимволов, сочетания операторов/ключевых слов, нехарактерные токены в параметрах, подозрительные последовательности кавычек/скобок).
- Система должна иметь возможность включать/выключать детект по зонам: query‑params, headers, cookies, JSON‑поля, form‑поля.

### FR‑06. Детект XSS
- Система должна выявлять XSS минимум по двум классам методов:
  - Сигнатурный детект (script‑теги, обработчики событий on*, javascript:, data: с HTML/JS, опасные функции/объекты и типовые payload‑паттерны).
  - Эвристика (контекст‑независимая оценка: HTML‑теги/атрибуты в параметрах, подозрительная энтити‑кодировка, смешанное кодирование, наличие шаблонов попытки закрыть контекст).
- Система должна поддерживать декодирование HTML entities и типовые обфускации (минимум: &lt;script&gt;, &#xNN;).

### FR‑07. Правила, политики и вердикты
- Система должна поддерживать политики обработки:
  - Allow (пропустить), Block (заблокировать), LogOnly (только записать).
- Система должна поддерживать приоритеты правил и порядок применения (например: whitelist перед blacklist).
- Система должна поддерживать исключения (allowlist) по: IP/CIDR, пути, параметру, заголовку, user‑agent, идентификатору правила.

### FR‑08. Оценка риска (скоринг)
- Система должна вычислять общий risk score запроса на основе срабатываний правил/эвристик.
- Система должна позволять задавать пороги принятия решений (например: score >= X -> block).

### FR‑09. Ответ при блокировке
- При блокировке система должна возвращать корректный HTTP‑ответ (по умолчанию 403) с настраиваемым телом (HTML/JSON) и заголовками.
- Система должна добавлять корреляционный идентификатор (request_id) в ответ и в журнал.

### FR‑10. Логирование и хранение событий
- Система должна создавать событие для каждого запроса с вердиктом, а также для подозрительных запросов — расширенное событие (с деталями срабатываний).
- Событие должно содержать: timestamp, request_id, client_ip, host, path, rule_id(ы), score, действие, краткое объяснение.
- Система должна поддерживать ротацию логов и настройку времени хранения.

### FR‑11. Поиск и фильтрация в веб‑панели
- Веб‑панель должна позволять фильтровать события по времени, host, пути, IP, вердикту, rule_id, score, меткам (false positive/true positive).
- Веб‑панель должна показывать детали: оригинальный и нормализованный запрос, сработавшие правила, объяснение.

### FR‑12. Управление правилами через веб‑панель
- Веб‑панель должна позволять создавать/редактировать/удалять правила:
  - Регулярные выражения/строковые сигнатуры.
  - Простые эвристики (ограничения на длины, энтропия, доля спецсимволов, запрет определённых токенов).
  - Составные правила (AND/OR, несколько зон анализа).
- Должна быть поддержка включения режима LogOnly для тестирования правил.

### FR‑13. Наблюдаемость (observability)
- Система должна предоставлять метрики: RPS, latency, число блокировок/срабатываний по правилам, top hosts/paths, доля 4xx/5xx.
- Система должна иметь health‑endpoint (например: /healthz) и readiness‑endpoint.

### FR‑14. Интеграции и уведомления
- Система должна поддерживать экспорт событий через API (REST).

### FR‑15. Управление конфигурацией
- Система должна поддерживать конфигурацию через файл (YAML/JSON) и горячую перезагрузку без простоя (graceful reload).
- Система должна поддерживать версионирование конфигурации и откат (минимум: хранение последних N версий).


## 6. Веб‑составляющая (минимальный функционал)
- Аутентификация в панели (логин/пароль).
- RBAC: минимум роли Admin и Analyst (разные права на изменение правил).
- Дашборд: графики блокировок/алертов, топ‑правил, топ‑путей.
- Страница «Правила»: список, статус, режим (Block/LogOnly), тестирование на примерах.
- Страница «События»: лента инцидентов, поиск, экспорт.
