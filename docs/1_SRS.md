# Техническое задания 

> Функциональные и нефункциональные требования к разрабатываемому ПО по части структуры спецификации "Software Requirements Specification". 

## 1 Функциональные требования

> Уточненный (по группам функциональных требований к проекту) перечень функций, выполняемых системой.

### Требования к управлению пользователями 
#### Функциональное требование 1

**Описание:** Система должна поддерживать создание учётных записей пользователей панели управления с ролями Admin и Analyst.
**Триггер запуска функции:** Администратор создаёт учётную запись через базу данных PostgreSQL (seed или ручное добавление).

#### Функциональное требование 2

**Описание:** Система должна хранить пароли пользователей в виде bcrypt-хэшей.
**Триггер запуска функции:** При создании или обновлении пароля пользователя.

#### Функциональное требование 3

**Описание:** Система должна поддерживать RBAC с двумя ролями:
- **Admin** — полный доступ (просмотр, создание, редактирование, удаление правил; просмотр и экспорт событий).
- **Analyst** — только чтение (просмотр правил, просмотр и экспорт событий).

**Триггер запуска функции:** При каждом запросе к защищённым эндпоинтам Admin API.

### Требования к аутентификации и авторизации

#### Функциональное требование 4

**Описание:** Система должна аутентифицировать пользователей через пару логин/пароль и выдавать JWT-токен (HS256) с TTL, настраиваемым в конфигурации (по умолчанию 24 часа).
**Триггер запуска функции:** POST-запрос на `/api/auth/login` с корректными учётными данными.

#### Функциональное требование 5

**Описание:** Все защищённые эндпоинты Admin API должны требовать валидный JWT-токен в заголовке `Authorization: Bearer <token>`. При невалидном или отсутствующем токене — ответ 401.
**Триггер запуска функции:** Каждый запрос к защищённому эндпоинту.

#### Функциональное требование 6

**Описание:** Эндпоинты модификации правил (POST, PUT, DELETE `/api/rules`) должны быть доступны только пользователям с ролью Admin. При недостаточных правах — ответ 403.
**Триггер запуска функции:** Запрос на модификацию правила от пользователя с ролью Analyst.

### Требования к функциям загрузки и обработки файлов

#### Функциональное требование 7

**Описание:** Система должна перехватывать и анализировать HTTP-запросы к защищаемому приложению, извлекая: метод, URI, query-string, заголовки, cookies, тело запроса (с ограничением по размеру `max_body_size`), IP/порт, User-Agent, Referer, Content-Type.
**Триггер запуска функции:** Автоматически при получении каждого HTTP-запроса через reverse-proxy.

#### Функциональное требование 8

**Описание:** Система должна нормализовать запросы перед детекцией: декодирование URL-encoding (до `max_decode_passes` проходов), унификация регистра, нормализация путей, удаление повторяющихся разделителей.
**Триггер запуска функции:** Автоматически на этапе normalize в middleware-цепочке.

#### Функциональное требование 9

**Описание:** Система должна выявлять SQL-инъекции сигнатурным (regex-паттерны: UNION SELECT, OR 1=1, stacked queries, time-based, boolean-based, DML/DDL) и эвристическим (доля спецсимволов, подозрительные последовательности) методами.
**Триггер запуска функции:** Автоматически на этапе detect в middleware-цепочке.

#### Функциональное требование 10

**Описание:** Система должна выявлять XSS-атаки сигнатурным (script-теги, обработчики событий on*, javascript: URI, data: URI, опасные теги, опасные функции) и эвристическим (HTML-теги в параметрах) методами.
**Триггер запуска функции:** Автоматически на этапе detect в middleware-цепочке.

#### Функциональное требование 11

**Описание:** Система должна вычислять risk score запроса как сумму весов сработавших правил и принимать решение на основе порогов: `log_threshold` (по умолчанию 3) — логирование, `block_threshold` (по умолчанию 7) — блокировка.
**Триггер запуска функции:** Автоматически после выполнения всех правил детекции.

#### Функциональное требование 12

**Описание:** Система должна поддерживать allowlist (белый список) по IP-адресам и путям для исключения из анализа.
**Триггер запуска функции:** Перед запуском детекции, на основе конфигурации `detection.allowlist`.

#### Функциональное требование 13

**Описание:** Система должна загружать правила детекции из YAML-файла (`rules.yaml`) и поддерживать управление правилами через Admin API (CRUD-операции) с хранением в PostgreSQL.
**Триггер запуска функции:** При старте системы (из файла) и при API-запросах на управление правилами.

#### Функциональное требование 14

**Описание:** Система должна записывать событие для каждого запроса с расширенными данными (timestamp, request_id, client_ip, host, path, rule_ids, score, verdict) в ClickHouse с батч-записью (размер батча и интервал flush настраиваются).
**Триггер запуска функции:** Автоматически после принятия решения по каждому запросу.

#### Функциональное требование 15

**Описание:** Система должна предоставлять Prometheus-метрики (RPS, latency, число блокировок) на отдельном порту `:9090` по пути `/metrics`.
**Триггер запуска функции:** При каждом HTTP-запросе через прокси (обновление счётчиков) и при scrape от Prometheus.

## 2 Требования к интерфейсам

> Уточненный список требований к интерфейсам системы.
### Требования к графическому интерфейсу управления администратора системы

1. Панель управления должна быть реализована как SPA на React 19 с TypeScript.
2. Интерфейс должен включать боковую панель навигации (Sidebar) с пунктами: Dashboard, Events, Rules.
3. Интерфейс должен поддерживать адаптивную верстку с использованием Tailwind CSS v4 и компонентов shadcn/ui.
4. Должна быть реализована страница авторизации с формой логин/пароль.
5. При отсутствии JWT-токена пользователь автоматически перенаправляется на страницу логина.

### Требования внешним интерфейсам системы HTTP API

1. Admin API должен быть доступен на порту `:8090`.
2. Proxy должен быть доступен на порту `:8080`.
3. Метрики Prometheus должны быть доступны на порту `:9090`.
4. API должен поддерживать CORS для origin `http://localhost:5173` и `http://localhost:3000`.
5. Все ответы API должны быть в формате `application/json`.
6. Ошибки API должны возвращаться в формате `{"error": "описание"}`.

### Требования к графическому интерфейсу пользователя системы

1. **Страница Dashboard** — графики блокировок/алертов за период, топ-правил по срабатываниям, топ-путей, общие метрики (RPS, всего событий, заблокировано, средний score).
2. **Страница Events** — таблица событий с фильтрацией по времени, IP, вердикту, rule_id; пагинация; переход к детальному просмотру события.
3. **Страница Event Detail** — полная информация о событии: оригинальный и нормализованный запрос, сработавшие правила, score, вердикт.
4. **Страница Rules** — список правил с отображением статуса (enabled/disabled), режима (Block/LogOnly), категории, веса; возможность создания, редактирования и удаления правил (для Admin).

## 3 Нефункциональные требования
> Уточненный список нефункциональных требований системе.
### Требования к обработке больших данных

1. ClickHouse используется как хранилище событий благодаря колоночному хранению и эффективным аналитическим запросам.
2. Батч-запись событий с настраиваемым размером батча (`batch_size: 100`) и интервалом flush (`flush_interval: 1s`) для оптимизации записи.
3. Лимит на количество возвращаемых событий через API: максимум 100 записей на страницу, максимум 10000 для экспорта.

### Требования к безопасности

1. Пароли хранятся в виде bcrypt-хэшей.
2. Аутентификация через JWT-токены с HS256 и настраиваемым TTL.
3. RBAC с ролями Admin и Analyst.
4. Ограничение размера тела запроса (`http.MaxBytesReader`) для предотвращения атак переполнения.
5. Валидация имени базы данных ClickHouse regex-паттерном для предотвращения SQL-инъекций в инфраструктурном слое.
6. Использование stdlib `regexp` (RE2) для линейного времени выполнения regex — защита от catastrophic backtracking при обработке вредоносного ввода.

### Требования к инфраструктуре

1. Развёртывание через docker-compose (dev и prod конфигурации).
2. Health check для всех сервисов (PostgreSQL: `pg_isready`, ClickHouse: `/ping`, WAF: `/healthz`, Admin API: `/api/health`).
3. В prod-конфигурации — persistent volumes для данных БД, политика `restart: unless-stopped`.
4. Конфигурация через YAML-файл (`config.yaml`) с поддержкой переопределения через переменные окружения.
