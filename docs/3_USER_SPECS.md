# Руководство пользователя

> Описание порядка взаимодействия пользователя поддерживаемых ролей (пользователь, администратор) с ПО. 
> Требования к структуре руководства пользователя по серии ГОСТ 34 устанавливаются ГОСТ Р 59795 - 2021.
# Описание операций, доступных пользователю
> Для каждой операции необходимо описать:
> - наименование;
> - условия, при соблюдении которых возможно выполнение операции;
> - подготовительные действия;
> - основные действия в требуемой последовательности;
> - заключительные действия.

## Операция 1: Авторизация пользователя в системе

1. **Наименование:**
   Авторизация пользователя в панели управления WAF.

2. **Условия выполнения:**
   - Система запущена и доступна (docker-compose up).
   - Администратор создал учётную запись пользователя в базе данных PostgreSQL.
   - Веб-панель доступна по адресу `http://localhost:5173`.

3. **Подготовительные действия:**
   - Открыть браузер (рекомендуется Chrome, Firefox или Edge последних версий).
   - Убедиться, что сервисы WAF и Admin API запущены и доступны.

4. **Основные действия:**
   - Перейти в браузере по адресу `http://localhost:5173`.
   - Система автоматически перенаправит на страницу авторизации `/login`.
   - В поле «Username» ввести имя пользователя.
   - В поле «Password» ввести пароль.
   - Нажать кнопку «Login».

5. **Заключительные действия:**
   При успешной авторизации пользователь будет перенаправлен на страницу Dashboard (`/dashboard`). JWT-токен сохраняется в хранилище браузера для последующих запросов.

## Операция 2: Просмотр дашборда

1. **Наименование:**
   Просмотр сводной информации о состоянии WAF на странице Dashboard.

2. **Условия выполнения:**
   - Пользователь авторизован в системе (роль Admin или Analyst).

3. **Подготовительные действия:**
   - Не требуются.

4. **Основные действия:**
   - В боковой панели навигации выбрать пункт «Dashboard».
   - На странице отображаются:
     - Графики блокировок и алертов за период.
     - Топ-правила по количеству срабатываний.
     - Топ-пути по количеству блокировок.
     - Общие метрики: RPS, количество событий, заблокировано, средний score.

5. **Заключительные действия:**
   Данные обновляются автоматически при повторном посещении страницы.

## Операция 3: Просмотр и фильтрация событий

1. **Наименование:**
   Просмотр ленты событий безопасности WAF с применением фильтров.

2. **Условия выполнения:**
   - Пользователь авторизован в системе (роль Admin или Analyst).

3. **Подготовительные действия:**
   - Не требуются.

4. **Основные действия:**
   - В боковой панели навигации выбрать пункт «Events».
   - На странице отображается таблица событий с колонками: время, IP, метод, путь, вердикт, score.
   - Для фильтрации событий использовать поля фильтрации:
     - **IP** — фильтр по IP-адресу клиента.
     - **Verdict** — фильтр по вердикту (allow, block, log_only).
     - **Rule ID** — фильтр по идентификатору сработавшего правила.
     - **From / To** — фильтр по временному диапазону.
   - Для перехода между страницами использовать пагинацию.
   - Для просмотра подробной информации о событии — нажать на строку в таблице.

5. **Заключительные действия:**
   При переходе к детальному просмотру отображается полная информация: оригинальный и нормализованный запрос, список сработавших правил, итоговый score и вердикт.

## Операция 4: Экспорт событий

1. **Наименование:**
   Экспорт событий безопасности по заданным фильтрам в формате JSON.

2. **Условия выполнения:**
   - Пользователь авторизован в системе (роль Admin или Analyst).

3. **Подготовительные действия:**
   - Настроить фильтры на странице Events или подготовить параметры экспорта.

4. **Основные действия:**
   - Отправить POST-запрос на `/api/events/export` с телом:
     ```json
     {
       "from": "2026-01-01T00:00:00Z",
       "to": "2026-04-09T23:59:59Z",
       "ip": "",
       "verdict": "block",
       "rule_id": "",
       "limit": 1000
     }
     ```
   - В ответ будет возвращён JSON-массив событий (до 10000 записей).

5. **Заключительные действия:**
   Полученные данные можно сохранить в файл для дальнейшего анализа.

## Операция 5: Управление правилами (Admin)

1. **Наименование:**
   Создание, редактирование и удаление правил детекции.

2. **Условия выполнения:**
   - Пользователь авторизован с ролью Admin.

3. **Подготовительные действия:**
   - Перейти на страницу «Rules» через боковую панель навигации.

4. **Основные действия:**
   - **Создание правила:**
     - Нажать кнопку создания нового правила.
     - Заполнить обязательные поля: Rule ID, Name, Type (regex/heuristic), Category (sqli/xss), Weight.
     - Для regex-правил указать Pattern (регулярное выражение).
     - Для эвристических правил указать Heuristic (имя функции) и Threshold.
     - Указать Targets (зоны анализа: query, body, headers, cookies, path).
     - Установить статус Enabled и режим (Block или LogOnly).
     - Сохранить правило.
   - **Редактирование правила:**
     - Нажать кнопку редактирования у нужного правила в списке.
     - Изменить необходимые поля.
     - Сохранить изменения.
   - **Удаление правила:**
     - Нажать кнопку удаления у нужного правила.
     - Подтвердить удаление.
   - **Переключение режима LogOnly:**
     - Для тестирования правила без блокировки включить режим LogOnly.

5. **Заключительные действия:**
   Изменения правил вступают в силу немедленно. Правила хранятся в PostgreSQL и доступны через Admin API.

## Операция 6: Просмотр правил (Analyst)

1. **Наименование:**
   Просмотр списка правил детекции без возможности модификации.

2. **Условия выполнения:**
   - Пользователь авторизован с ролью Analyst.

3. **Подготовительные действия:**
   - Не требуются.

4. **Основные действия:**
   - В боковой панели навигации выбрать пункт «Rules».
   - На странице отображается таблица правил с колонками: ID, Name, Type, Category, Weight, Enabled, LogOnly.
   - Для просмотра деталей правила нажать на строку в таблице.

5. **Заключительные действия:**
   Пользователь с ролью Analyst может только просматривать правила. Кнопки создания, редактирования и удаления недоступны.
